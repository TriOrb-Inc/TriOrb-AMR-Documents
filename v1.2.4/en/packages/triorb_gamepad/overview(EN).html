Overview¶

Package that provides a ROS 2 node for safely controlling a TriOrb AMR from commercial or dedicated gamepads. It monitors /dev/input/js*, detects supported devices automatically, and generates drive, lifter, and emergency-stop commands according to button and stick mappings defined in JSON files. Node and topic names are prefixed by ROS_PREFIX to avoid collisions.

Main Features¶

  • Convert joystick input into TriOrb AMR velocity, lifter, and stop commands

  • Switch button/axis assignments and robot-specific parameters through JSON dictionaries

  • Automatically switch between normal, SafeRun, and collaboration modes according to subscriber state

Notes¶

Provided Node¶

gamepad¶

  • Implemented as a single node in the _Node class inside src/triorb_gamepad.cpp. Because UNIQUE_NODE is enabled, startup exits automatically if another node with the same name already exists.

  • When the dev parameter is empty, the node scans /dev/input/js0..js15 and uses the first responsive device.

  • Gamepad information such as device name, axis count, and button count is read at startup, then the node selects a mapping from either a user dictionary or a preset under share/triorb_gamepad/params.

  • The event loop accumulates velocity, acceleration, and lifter state in the AMRCtrl structure and only publishes a TriOrb RunVel command when the state changes.

Mode Management¶

  • A 1 s timer checks subscriber counts. If /collab/wakeup has subscribers, the node switches automatically to collaboration mode and publishes to /collab/... topics.

  • safe_run_mode depends on whether safe_drive/run_vel has subscribers or the node was already in SafeRun mode.

  • active and inactive commands can explicitly enable or disable the device. When inactive, velocity is fixed at zero.

Operational Notes¶

  • The node is implemented as a lifecycle node. With the default auto_activate:=false, it waits in unconfigured until configure() and activate() are requested externally.

  • If a device is missing or read errors such as ENODEV / EIO occur, the node exits the current device session and retries after reporting an error.

  • If need_disconnect() stays true because invalid values keep arriving, the device is closed and re-detected.
